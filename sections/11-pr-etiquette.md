**One PR, One Purpose.** A PR that touches one thing is faster to review, easier to approve, and easier to revert if something goes wrong.
[/CALLOUT]

### As a contributor, your PRs should:

- Write clear PR descriptions explaining *what* changed and *why*
- Keep PRs small and focused — one new feature, one bug fix, or one refactor
- Respond to feedback graciously and **don't take it personally**
- If the PR isn't approved and you get feedback, make edits by pushing new commits to the same branch — the PR updates automatically. You shouldn't be opening a new PR.

### As an admin or maintainer approving PRs:

- Look at the diff. Does the change actually do what the PR description says it does?
- Check for obvious bugs, typos, or unintended side effects
- Make sure the change doesn't break anything that was already working
- Be specific with feedback — "this function could be simplified" is not very helpful. Point to the exact line.
- Be kind. The goal is to improve the code, not criticize the person
- Ask questions rather than making demands — "what do you think about...?" rather than "change this to..."
- You can approve a PR with comments for non-blocking suggestions

[CALLOUT type="note"]
This lesson will not focus on how admins or maintainers check and test code in depth. It's encouraged that you look up resources to learn more about code review practices.
[/CALLOUT]
