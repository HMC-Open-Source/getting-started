**One PR, One Purpose**

A PR should only touch on ONE new feature, ONE bug fix, OR ONE change to help refactor the code. It's just easier for reviewers to understand what's going on if you keep it focused. This doesn't mean one file — a single feature might naturally touch multiple files, and that's fine. What matters is that everything in the PR serves the same purpose.

**As a contributor, your PRs should:**

- Write clear PR descriptions explaining *what* changed and *why*
- Keep PRs small and focused — one purpose per PR
- Respond to feedback graciously and **don't take it personally**
- If the PR isn't approved and you get feedback, make edits by pushing new commits to the same branch. You shouldn't be opening a new PR — the existing one updates automatically

**As an admin or maintainer approving PRs:**

- Look at the diff. Does the change actually do what the PR description says it does?
- Check for obvious bugs, typos, or unintended side effects
- Make sure the change doesn't break anything that was already working
- Be specific with feedback. Saying "this function could be simplified" is not very helpful — point to the exact line
- Be kind. The goal is to improve the code, not criticize the person
- Ask questions rather than making demands — "what do you think about...?" rather than "change this to..."
- You can approve a PR with comments for non-blocking suggestions

**Note:** This zine will not focus on how admins or maintainers check and test code. It's encouraged that you look up resources to learn more about this part of the process.
